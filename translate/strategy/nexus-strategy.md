## 1. 战略基础

### 1.1 NEXUS解决什么问题

单个代理很强大。但缺乏协调时，它们会产生：
- 冲突的架构决策
- 跨部门的重复工作
- 交接边界的质量缺口
- 无共享上下文或机构记忆

**NEXUS消除这些失败模式**，通过定义：
- **谁**在每个阶段激活
- **什么**他们生产以及为谁生产
- **何时**他们交接以及交给谁
- **如何**在推进前验证质量
- **为什么**每个代理存在于管道中（无乘客）

### 1.2 核心原则

| 原则 | 描述 |
|------|------|
| **管道完整性** | 不通过质量门不得进入下一阶段 |
| **上下文连续性** | 每次交接携带完整上下文——无代理从零开始 |
| **并行执行** | 独立工作流并发运行以压缩时间线 |
| **证据优于声明** | 所有质量评估需要证明，而非断言 |
| **快速失败，快速修复** | 每个任务最多3次重试后升级 |
| **单一真相来源** | 一个规范规范、一个任务列表、一个架构文档 |

### 1.3 按部门划分的代理名册

| 部门 | 代理 | 主要NEXUS角色 |
|------|------|---------------|
| **工程** | 前端开发者、后端架构师、移动应用构建者、AI工程师、DevOps自动化者、快速原型师、高级开发者 | 构建、部署和维护所有技术系统 |
| **设计** | UI设计师、UX研究员、UX架构师、品牌守护者、视觉叙事者、趣味注入者、图像提示工程师 | 定义视觉身份、用户体验和品牌一致性 |

## 2. NEXUS运行模型

### 2.1 七阶段管道

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        NEXUS管道                                        │
│                                                                         │
│  阶段0          阶段1           阶段2           阶段3                    │
│  发现     ───▶  策略化    ───▶  脚手架    ───▶  构建                    │
│  智能           架构           基础            开发↔QA循环               │
│                                                                         │
│  阶段4          阶段5           阶段6                                    │
│  加固     ───▶  发布      ───▶  运营                                     │
│  质量门         上市            持续运营                                  │
│                                                                         │
│  ◆ 每个阶段之间有质量门                                                 │
│  ◆ 阶段内有并行轨道                                                      │
│  ◆ 每个边界有反馈循环                                                    │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 指挥结构

```
                    ┌──────────────────────┐
                    │  代理编排者           │  ◄── 管道控制器
                    │  (专业部门)           │
                    └──────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
     ┌────────▼──────┐ ┌──────▼───────┐ ┌──────▼──────────┐
     │ 工作室        │ │ 项目         │ │ 高级项目        │
     │ 制作人        │ │ 牧羊人       │ │ 经理            │
     │ (组合)        │ │ (执行)       │ │ (任务范围)      │
     └───────────────┘ └──────────────┘ └─────────────────┘
              │                │                │
              ▼                ▼                ▼
     ┌─────────────────────────────────────────────────┐
     │           部门负责人 (每个阶段)                   │
     │  工程 │ 设计 │ 营销 │ 产品 │ QA                 │
     └─────────────────────────────────────────────────┘
```

### 2.3 激活模式

NEXUS支持三种部署配置：

| 模式 | 活跃代理 | 用例 | 时间线 |
|------|----------|------|--------|
| **NEXUS-Full** | 全部 | 企业产品发布，完整生命周期 | 12-24周 |
| **NEXUS-Sprint** | 15-25 | 功能开发，MVP构建 | 2-6周 |
| **NEXUS-Micro** | 5-10 | Bug修复，内容活动，单一交付物 | 1-5天 |

## 3. 阶段0 — 智能与发现

> **目标**：在投入资源前了解环境。问题验证前不构建。

### 3.1 活跃代理

| 代理 | 阶段角色 | 主要输出 |
|------|----------|----------|
| **趋势研究员** | 市场情报负责人 | 包含TAM/SAM/SOM的市场分析报告 |
| **反馈综合者** | 用户需求分析 | 包含痛点的综合反馈报告 |
| **UX研究员** | 用户行为分析 | 包含人物画像和旅程图的研究发现 |
| **分析报告员** | 数据环境评估 | 包含可用信号的数据审计报告 |
| **法律合规检查者** | 监管扫描 | 合规要求矩阵 |
| **工具评估者** | 技术环境 | 技术栈评估 |

### 3.2 并行工作流

```
工作流A：市场情报                    工作流B：用户情报
├── 趋势研究员                        ├── 反馈综合者
│   ├── 竞争格局                      │   ├── 多渠道反馈收集
│   ├── 市场规模(TAM/SAM/SOM)         │   ├── 情感分析
│   └── 趋势生命周期映射              │   └── 痛点优先级排序
│                                     │
├── 分析报告员                        ├── UX研究员
│   ├── 现有数据审计                  │   ├── 用户访谈/调查
│   ├── 信号识别                      │   ├── 人物画像开发
│   └── 基线指标                      │   └── 旅程映射
│                                     │
└── 法律合规检查者                    └── 工具评估者
    ├── 监管要求                          ├── 技术评估
    ├── 数据处理约束                      ├── 自建vs购买分析
    └── 管辖区映射                        └── 集成可行性
```

### 3.3 阶段0质量门

**门卫**：执行摘要生成器

| 标准 | 阈值 | 所需证据 |
|------|------|----------|
| 市场机会验证 | TAM > 最低可行阈值 | 趋势研究员报告及来源 |
| 用户需求确认 | ≥3个验证的痛点 | 反馈综合者 + UX研究员数据 |
| 监管路径清晰 | 无阻塞性合规问题 | 法律合规检查者矩阵 |
| 数据基础评估 | 关键指标已识别 | 分析报告员审计 |
| 技术可行性确认 | 技术栈已验证 | 工具评估者评估 |

**输出**：执行摘要（≤500字，SCQA格式）→ 决策：通过 / 不通过 / 转向

## 4. 阶段1 — 策略与架构

> **目标**：定义我们构建什么、如何构建以及成功标准——在编写一行代码之前。

### 4.1 活跃代理

| 代理 | 阶段角色 | 主要输出 |
|------|----------|----------|
| **工作室制作人** | 战略组合对齐 | 战略组合计划 |
| **高级项目经理** | 规范到任务转换 | 全面任务列表 |
| **冲刺优先者** | 功能优先级排序 | 优先级待办事项（RICE评分） |
| **UX架构师** | 技术架构 + UX基础 | 架构规范 + CSS设计系统 |
| **品牌守护者** | 品牌身份系统 | 品牌基础文档 |
| **后端架构师** | 系统架构 | 系统架构规范 |
| **AI工程师** | AI/ML架构（如适用） | ML系统设计 |
| **财务追踪者** | 预算和资源规划 | 包含ROI预测的财务计划 |

### 4.2 执行顺序

```
步骤1：战略框架（并行）
├── 工作室制作人 → 战略组合计划（愿景、目标、ROI目标）
├── 品牌守护者 → 品牌基础（目的、价值观、视觉身份系统）
└── 财务追踪者 → 预算框架（资源分配、成本预测）

步骤2：技术架构（并行，步骤1后）
├── UX架构师 → CSS设计系统 + 布局框架 + UX结构
├── 后端架构师 → 系统架构（服务、数据库、API）
├── AI工程师 → ML架构（模型、管道、推理策略）
└── 高级项目经理 → 任务列表（规范 → 任务，确切需求）

步骤3：优先级排序（顺序，步骤2后）
└── 冲刺优先者 → RICE评分待办事项及冲刺分配
    ├── 输入：任务列表 + 架构规范 + 预算框架
    ├── 输出：带依赖图的优先冲刺计划
    └── 验证：工作室制作人确认战略对齐
```

### 4.3 阶段1质量门

**门卫**：工作室制作人 + 现实检查者（双重签署）

| 标准 | 阈值 | 所需证据 |
|------|------|----------|
| 架构覆盖所有需求 | 100%规范覆盖 | 高级PM任务列表交叉引用 |
| 品牌系统完整 | Logo、颜色、排版、语调已定义 | 品牌守护者交付物 |
| 技术可行性验证 | 所有组件有实现路径 | 后端架构师 + UX架构师规范 |
| 预算批准 | 在组织约束内 | 财务追踪者计划 |
| 冲刺计划现实 | 基于速度估算 | 冲刺优先者待办事项 |

**输出**：批准的架构包 → 阶段2激活

## 6. 阶段3 — 构建与迭代

> **目标**：通过持续开发↔QA循环实现功能。每个任务在下一个开始前验证。这是大部分工作发生的地方。

### 6.1 开发↔QA循环

这是NEXUS的核心。代理编排者管理**逐任务质量循环**：

```
┌─────────────────────────────────────────────────────────┐
│                   开发 ↔ QA 循环                         │
│                                                          │
│  ┌──────────┐    ┌──────────┐    ┌──────────────────┐   │
│  │ 开发者    │───▶│ 证据     │───▶│ 决策逻辑          │   │
│  │ 代理      │    │ 收集者   │    │                   │   │
│  │           │    │ (QA)     │    │ 通过 → 下一任务   │   │
│  │ 实现      │    │          │    │ 失败 → 重试(≤3)   │   │
│  │ 任务N     │    │ 测试     │    │ 阻塞 → 升级       │   │
│  │           │◀───│ 任务N    │◀───│                   │   │
│  └──────────┘    └──────────┘    └──────────────────┘   │
│       ▲                                    │             │
│       │            QA反馈                  │             │
│       └────────────────────────────────────┘             │
│                                                          │
│  编排者追踪：尝试次数、QA反馈、                             │
│  任务状态、累积质量指标                                    │
└─────────────────────────────────────────────────────────┘
```

## 11. 交接协议

### 11.1 标准交接文档格式

每次代理到代理的交接必须包含：

```markdown
## NEXUS交接文档

### 元数据
- **来自**：[代理名称] ([部门])
- **到**：[代理名称] ([部门])
- **阶段**：[当前NEXUS阶段]
- **任务引用**：[冲刺优先者待办事项中的任务ID]
- **优先级**：[关键 / 高 / 中 / 低]
- **时间戳**：[ISO 8601]

### 上下文
- **项目**：[项目名称和简要描述]
- **当前状态**：[到目前为止完成了什么]
- **相关文件**：[要审查的文件/制品列表]
- **依赖项**：[此工作依赖什么]

### 交付物请求
- **需要什么**：[具体、可衡量的交付物]
- **验收标准**：[如何衡量成功]
- **约束**：[技术、时间线或资源约束]
```

## 12. 质量门

### 12.1 门摘要

| 阶段 | 门名称 | 门卫 | 通过标准 |
|------|--------|------|----------|
| 0 → 1 | 发现门 | 执行摘要生成器 | 市场验证、用户需求确认、监管路径清晰 |
| 1 → 2 | 架构门 | 工作室制作人 + 现实检查者 | 架构完整、品牌定义、预算批准、冲刺计划现实 |
| 2 → 3 | 基础门 | DevOps自动化者 + 证据收集者 | CI/CD工作、骨架应用运行、监控活跃 |
| 3 → 4 | 功能门 | 代理编排者 | 所有任务通过QA、无关键bug、性能基线达标 |
| 4 → 5 | 生产门 | 现实检查者（唯一权威） | 用户旅程完整、跨设备一致、安全验证、规范合规 |
| 5 → 6 | 发布门 | 工作室制作人 + 分析报告员 | 部署成功、系统稳定、增长渠道活跃 |

### 12.2 门失败处理

```
如果门失败：
  ├── 门卫生成具体失败报告
  ├── 代理编排者将失败路由给负责代理
  ├── 失败项目进入开发↔QA循环（阶段3机制）
  ├── 最多3次门重试后升级到工作室制作人
  └── 工作室制作人决定：修复、降级或接受风险
```
