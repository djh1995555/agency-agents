# 🛡️ 第4阶段剧本 — 质量与加固

> **持续时间**：3-7天 | **代理**：8个 | **门控者**：现实检查器（唯一权威）

---

## 目标

最终质量考验。现实检查器默认"需要改进" — 你必须用压倒性的证据证明生产就绪。此阶段存在是因为首次实现通常需要2-3个修订周期，这是健康的。

## 前置条件

- [ ] 第3阶段质量门已通过（所有任务已QA）
- [ ] 第3阶段交接包已接收
- [ ] 所有功能已实现并单独验证

## 关键心态

> **现实检查器的默认裁决是需要改进。**
> 
> 这不是悲观 — 这是现实。生产就绪需要：
> - 完整用户旅程端到端工作
> - 跨设备一致性（桌面、平板、移动）
> - 负载下的性能（不只是正常路径）
> - 安全验证（不只是"我们添加了认证"）
> - 规范合规（每个需求，而非大多数）
>
> 首次通过获得B/B+评分是正常且预期的。

## 代理激活序列

### 步骤1：证据收集（第1-2天，全部并行）

#### 📸 证据收集器 — 全面视觉证据
```
激活证据收集器进行 [项目] 的全面系统证据收集。

所需交付物：
1. 完整截图套件：
   - 桌面（1920x1080）— 每个页面/视图
   - 平板（768x1024）— 每个页面/视图
   - 移动（375x667）— 每个页面/视图
2. 交互证据：
   - 导航流程（点击前后）
   - 表单交互（空、已填、已提交、错误状态）
   - 模态/对话框交互
   - 手风琴/可展开内容
3. 主题证据：
   - 浅色模式 — 所有页面
   - 深色模式 — 所有页面
   - 系统偏好检测
4. 错误状态证据：
   - 404页面
   - 表单验证错误
   - 网络错误处理
   - 空状态

格式：带test-results.json的截图证据包
时间表：2天
```

#### 🔌 API测试器 — 完整API回归
```
激活API测试器进行 [项目] 的完整API回归测试。

所需交付物：
1. 端点回归套件：
   - 所有端点测试（GET、POST、PUT、DELETE）
   - 认证/授权验证
   - 输入验证测试
   - 错误响应验证
2. 集成测试：
   - 跨服务通信
   - 数据库操作验证
   - 外部API集成
3. 边缘情况测试：
   - 速率限制行为
   - 大负载处理
   - 并发请求处理
   - 格式错误输入处理

格式：每个端点通过/失败的API测试报告
时间表：2天
```

#### ⚡ 性能基准测试器 — 负载测试
```
激活性能基准测试器进行 [项目] 的负载测试。

所需交付物：
1. 10倍预期流量的负载测试：
   - 响应时间分布（P50、P95、P99）
   - 负载下吞吐量
   - 负载下错误率
   - 资源利用率（CPU、内存、网络）
2. 核心Web指标测量：
   - LCP（最大内容绘制）< 2.5s
   - FID（首次输入延迟）< 100ms
   - CLS（累积布局偏移）< 0.1
3. 数据库性能：
   - 查询执行时间
   - 连接池利用率
   - 索引有效性
4. 压力测试结果：
   - 断点识别
   - 优雅降级行为
   - 过载后恢复时间

格式：性能认证报告
时间表：2天
```

#### ⚖️ 法律合规检查器 — 最终合规审计
```
激活法律合规检查器进行 [项目] 的最终合规审计。

所需交付物：
1. 隐私合规验证：
   - 隐私政策准确性
   - 同意管理功能
   - 数据主体权利实现
   - Cookie同意实现
2. 安全合规：
   - 数据加密（静态和传输中）
   - 认证安全
   - 输入清理
   - OWASP Top 10检查
3. 监管合规：
   - GDPR要求（如适用）
   - CCPA要求（如适用）
   - 行业特定要求
4. 无障碍合规：
   - WCAG 2.1 AA验证
   - 屏幕阅读器兼容性
   - 键盘导航

格式：合规认证报告
时间表：2天
```

### 步骤2：分析（第3-4天，并行，步骤1后）

#### 📊 测试结果分析器 — 质量指标聚合
```
激活测试结果分析器进行 [项目] 的质量指标聚合。

输入：所有步骤1报告
所需交付物：
1. 聚合质量仪表板：
   - 整体质量评分
   - 类别细分（视觉、功能、性能、安全、合规）
   - 问题严重程度分布
   - 趋势分析（如多次测试周期）
2. 问题优先级排序：
   - 关键问题（生产前必须修复）
   - 高优先级问题（生产前应该修复）
   - 中等问题（下个冲刺修复）
   - 低优先级问题（积压）
3. 风险评估：
   - 生产就绪概率
   - 剩余风险区域
   - 建议缓解措施

格式：质量指标仪表板
时间表：1天
```

#### 🔄 工作流优化器 — 流程效率审查
```
激活工作流优化器进行 [项目] 的流程效率审查。

输入：第3阶段执行数据 + 步骤1发现
所需交付物：
1. 流程效率分析：
   - 开发↔QA循环效率（首次通过率、平均重试次数）
   - 瓶颈识别
   - 不同问题类型的解决时间
2. 改进建议：
   - 第6阶段运营的流程变更
   - 自动化机会
   - 质量改进建议

格式：优化建议报告
时间表：1天
```

#### 🏗️ 基础设施维护者 — 生产就绪检查
```
激活基础设施维护者进行 [项目] 的生产就绪检查。

所需交付物：
1. 生产环境验证：
   - 所有服务健康且响应
   - 自动扩展已配置并测试
   - 负载均衡器配置已验证
   - SSL/TLS证书有效
2. 监控验证：
   - 所有关键指标正在收集
   - 警报规则已配置并测试
   - 仪表板访问已验证
   - 日志聚合工作正常
3. 灾难恢复验证：
   - 备份系统运营
   - 恢复程序已文档化并测试
   - 故障转移机制已验证
4. 安全验证：
   - 防火墙规则已审查
   - 访问控制已验证
   - 密钥管理已确认
   - 漏洞扫描清洁

格式：基础设施就绪报告
时间表：1天
```

### 步骤3：最终裁决（第5-7天，顺序执行）

#### 🔍 现实检查器 — 最终裁决
```
激活现实检查器进行 [项目] 的最终集成测试。

强制流程 — 不要跳过：

步骤1：现实检查命令
- 验证实际构建的内容（ls、grep声称的功能）
- 交叉检查声称的功能与规范
- 运行全面截图捕获
- 审查步骤1和步骤2的所有证据

步骤2：QA交叉验证
- 审查证据收集器发现
- 与API测试器结果交叉参考
- 验证性能基准测试器数据
- 确认法律合规检查器发现

步骤3：端到端系统验证
- 测试完整用户旅程（而非单个功能）
- 验证所有设备的响应行为
- 端到端检查交互流程
- 审查实际性能数据

步骤4：规范现实检查
- 引用原始规范的确切文本
- 与实际实现证据比较
- 记录规范与现实之间的每个差距
- 不做假设 — 仅证据

裁决选项：
- 就绪：压倒性的生产就绪证据（首次通过罕见）
- 需要改进：已识别具体问题及修复列表（预期）
- 未就绪：需要重新审视第1/2阶段的主要架构问题

格式：基于现实的集成报告
默认：需要改进，除非另有证明
```

## 质量门 — 最终门

| # | 标准 | 阈值 | 所需证据 |
|---|------|------|----------|
| 1 | 用户旅程完整 | 所有关键路径端到端工作 | 现实检查器截图 |
| 2 | 跨设备一致性 | 桌面 + 平板 + 移动都工作 | 响应式截图 |
| 3 | 性能认证 | P95 < 200ms、LCP < 2.5s、正常运行时间 > 99.9% | 性能基准测试器报告 |
| 4 | 安全验证 | 零关键漏洞 | 安全扫描 + 合规报告 |
| 5 | 合规认证 | 所有监管要求满足 | 法律合规检查器报告 |
| 6 | 规范合规 | 100%规范需求已实现 | 逐点验证 |
| 7 | 基础设施就绪 | 生产环境已验证 | 基础设施维护者报告 |

## 门决策

**唯一权威**：现实检查器

### 如果就绪（进入第5阶段）：
```markdown
## 第4阶段 → 第5阶段交接包

### 给发布团队：
- 现实检查器认证报告
- 性能认证
- 合规认证
- 基础设施就绪报告
- 已知限制（如有）

### 给增长黑客：
- 产品准备好迎接用户
- 营销消息的功能列表
- 可信度性能数据

### 给DevOps自动化器：
- 生产部署已批准
- 蓝绿部署计划
- 回滚程序已确认
```

### 如果需要改进（返回第3阶段）：
```markdown
## 第4阶段 → 第3阶段返回包

### 修复列表（来自现实检查器）：
1. [关键问题1]：[描述 + 证据 + 修复说明]
2. [关键问题2]：[描述 + 证据 + 修复说明]
3. [高优先级问题1]：[描述 + 证据 + 修复说明]
...

### 流程：
- 问题进入开发↔QA循环（第3阶段机制）
- 每个修复必须通过证据收集器QA
- 所有修复完成时 → 返回第4阶段步骤3
- 现实检查器用更新证据重新评估

### 预期：2-3个修订周期是正常的
```

### 如果未就绪（返回第1/2阶段）：
```markdown
## 第4阶段 → 第1/2阶段返回包

### 已识别的架构问题：
1. [根本问题]：[为何无法在第3阶段修复]
2. [结构性问题]：[架构层面需要改变什么]

### 建议行动：
- [ ] 修订系统架构（第1阶段）
- [ ] 重建基础（第2阶段）
- [ ] 缩减范围并重新定义（第1阶段）

### 需要工作室制作人决策
```

---

*第4阶段在现实检查器发布带压倒性证据的就绪裁决时完成。需要改进是预期的首次通过结果 — 这意味着系统工作但需要打磨。*
