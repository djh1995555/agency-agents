# 🔨 第3阶段剧本 — 构建与迭代

> **持续时间**：2-12周（因范围而异） | **代理**：15-30+ | **门控者**：代理编排器

## 目标

通过持续的开发↔QA循环实现所有功能。每个任务在下一个开始前都经过验证。这是大部分工作发生的地方 — 也是NEXUS编排产生最大价值的地方。

## 前置条件

- [ ] 第2阶段质量门已通过（基础已验证）
- [ ] 冲刺优先排序器积压可用，带RICE评分
- [ ] CI/CD管道运营
- [ ] 设计系统和组件库就绪
- [ ] API脚手架及认证系统就绪

## 开发↔QA循环 — 核心机制

代理编排器通过此循环管理每个任务：

```
对于冲刺积压中的每个任务（按RICE评分排序）：

  1. 将任务分配给适当的开发者代理（参见分配矩阵）
  2. 开发者实现任务
  3. 证据收集器测试任务
     - 视觉截图（桌面、平板、移动）
     - 根据验收标准进行功能验证
     - 品牌一致性检查
  4. 如果裁决 == 通过：
       标记任务完成
       移至下一个任务
     否则如果裁决 == 失败 且尝试次数 < 3：
       发送QA反馈给开发者
       开发者修复具体问题
       返回步骤3
     否则如果尝试次数 >= 3：
       升级给代理编排器
       编排器决定：重新分配、分解、推迟或接受
  5. 更新管道状态报告
```

## 冲刺执行模板

### 冲刺计划（第1天）

```
冲刺优先排序器激活：
1. 审查带更新RICE评分的积压
2. 根据团队速度选择冲刺任务
3. 将任务分配给开发者代理
4. 识别依赖和排序
5. 设定冲刺目标和成功标准

输出：带任务分配的冲刺计划
```

### 每日执行（第2天到第N-1天）

```
代理编排器管理：
1. 当前任务状态检查
2. 开发↔QA循环执行
3. 阻塞识别和解决
4. 进度跟踪和报告

状态报告格式：
- 今日完成任务：[列表]
- QA中的任务：[列表]
- 开发中的任务：[列表]
- 阻塞的任务：[列表及原因]
- QA通过率：[X/Y]
```

### 冲刺审查（第N天）

```
项目牧羊人主持：
1. 演示已完成的功能
2. 审查每个任务的QA证据
3. 收集利益相关者反馈
4. 根据学习更新积压

参与者：所有活跃代理 + 利益相关者
输出：冲刺审查摘要
```

### 冲刺回顾

```
工作流优化器主持：
1. 什么做得好？
2. 什么可以改进？
3. 下个冲刺我们要改变什么？
4. 流程效率指标

输出：回顾行动项
```

## 编排器决策逻辑

### 任务失败处理

```
当任务QA失败时：
  如果尝试次数 == 1：
    → 发送具体QA反馈给开发者
    → 开发者只修复已识别的问题
    → 重新提交QA
    
  如果尝试次数 == 2：
    → 发送累积QA反馈
    → 考虑：开发者代理是否合适？
    → 开发者带额外上下文修复
    → 重新提交QA
    
  如果尝试次数 == 3：
    → 升级
    → 选项：
      a) 重新分配给不同的开发者代理
      b) 将任务分解为更小的子任务
      c) 修订方法/架构
      d) 接受已知限制（记录）
      e) 推迟到未来冲刺
    → 记录决策和理由
```

### 并行任务管理

```
当多个任务无依赖时：
  → 同时分配给不同的开发者代理
  → 每个运行独立的开发↔QA循环
  → 编排器同时跟踪所有循环
  → 按依赖顺序合并已完成任务

当任务有依赖时：
  → 等待依赖通过QA
  → 然后分配依赖任务
  → 在交接中包含依赖上下文
```
