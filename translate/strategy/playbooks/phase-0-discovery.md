# 🔍 第0阶段剧本 — 情报与发现

> **持续时间**：3-7天 | **代理**：6个 | **门控者**：执行摘要生成器

---

## 目标

在投入资源之前验证机会。在理解问题、市场和监管环境之前不进行构建。

## 前置条件

- [ ] 项目简报或初始概念存在
- [ ] 利益相关者发起人已确定
- [ ] 发现阶段预算已批准

## 代理激活序列

### 第一波：并行启动（第1天）

#### 🔍 趋势研究员 — 市场情报负责人
```
激活趋势研究员进行 [项目领域] 的市场情报工作。

所需交付物：
1. 竞争格局分析（直接 + 间接竞争对手）
2. 市场规模：TAM、SAM、SOM及方法论
3. 趋势生命周期映射：此市场处于采用曲线的哪个位置？
4. 3-6个月趋势预测及置信区间
5. 该领域的投资和融资趋势

来源：最少15个独特、已验证的来源
格式：带执行摘要的战略报告
时间表：3天
```

#### 💬 反馈综合器 — 用户需求分析
```
激活反馈综合器进行 [项目领域] 的用户需求分析。

所需交付物：
1. 多渠道反馈收集计划（调查、访谈、评论、社交）
2. 现有用户触点的情感分析
3. 痛点识别和优先级排序（RICE评分）
4. 功能请求分析及商业价值估算
5. 从反馈模式中识别流失风险指标

格式：带优先级矩阵的综合反馈报告
时间表：3天
```

#### 🔍 UX研究员 — 用户行为分析
```
激活UX研究员进行 [项目领域] 的用户行为分析。

所需交付物：
1. 用户访谈计划（5-10个目标用户）
2. 人物画像开发（3-5个主要人物画像）
3. 主要用户流程的旅程映射
4. 竞争产品的可用性启发式评估
5. 带统计验证的行为洞察

格式：带人物画像和旅程地图的研究发现报告
时间表：5天
```

### 第二波：并行启动（第1天，独立于第一波）

#### 📊 分析报告器 — 数据环境评估
```
激活分析报告器进行 [项目领域] 的数据环境评估。

所需交付物：
1. 现有数据源审计（有哪些数据可用？）
2. 信号识别（我们可以测量什么？）
3. 基线指标建立
4. 数据质量评估及完整性评分
5. 分析基础设施建议

格式：带信号图的数据审计报告
时间表：2天
```

#### ⚖️ 法律合规检查器 — 监管扫描
```
激活法律合规检查器进行 [项目领域] 的监管扫描。

所需交付物：
1. 适用监管框架（GDPR、CCPA、HIPAA等）
2. 数据处理要求和约束
3. 目标市场的司法管辖区映射
4. 合规风险评估及严重程度评级
5. 阻塞性 vs 可管理的合规问题

格式：合规要求矩阵
时间表：3天
```

#### 🛠️ 工具评估器 — 技术环境
```
激活工具评估器进行 [项目领域] 的技术环境评估。

所需交付物：
1. 问题领域的技术栈评估
2. 关键组件的自建 vs 购买分析
3. 与现有系统的集成可行性
4. 开源 vs 商业评估
5. 技术风险评估

格式：带推荐矩阵的技术栈评估
时间表：2天
```

## 汇聚点（第5-7天）

所有六个代理交付报告。执行摘要生成器综合：

```
激活执行摘要生成器综合第0阶段发现。

输入文档：
1. 趋势研究员 → 市场分析报告
2. 反馈综合器 → 综合反馈报告
3. UX研究员 → 研究发现报告
4. 分析报告器 → 数据审计报告
5. 法律合规检查器 → 合规要求矩阵
6. 工具评估器 → 技术栈评估

输出：执行摘要（≤500字，SCQA格式）
需要决策：推进 / 不推进 / 转型
包含：量化市场机会、已验证用户需求、监管路径、技术可行性
```

## 质量门清单

| # | 标准 | 证据来源 | 状态 |
|---|------|----------|------|
| 1 | 市场机会已验证，TAM > 最低可行阈值 | 趋势研究员报告 | ☐ |
| 2 | ≥3个已验证用户痛点，有支持数据 | 反馈综合器 + UX研究员 | ☐ |
| 3 | 未识别阻塞性合规问题 | 法律合规检查器矩阵 | ☐ |
| 4 | 关键指标和数据源已识别 | 分析报告器审计 | ☐ |
| 5 | 技术栈可行且已评估 | 工具评估器评估 | ☐ |
| 6 | 执行摘要已交付，带推进/不推进建议 | 执行摘要生成器 | ☐ |

## 门决策

- **推进**：进入第1阶段 — 战略与架构
- **不推进**：归档发现，记录学习，重新分配资源
- **转型**：根据发现修改范围/方向，重新运行针对性发现
